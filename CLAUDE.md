# CLAUDE.md — Claude Code guide for 595BowersHub

This project is developed with **two agentic tools that share one repo**: the **Kiro IDE** (spec-driven, AWS) and **Claude Code** (this CLI). Kiro is the primary tool; Claude Code works alongside it. This file orients Claude Code — it deliberately does **not** duplicate project context, it points at the canonical sources.

## Working alongside Kiro — coexistence protocol

- **Commit at each handoff.** Before switching between Kiro and Claude Code, commit (or stash) so the other tool starts from a clean tree and the diff of what each did is legible. Don't run both editing the same files simultaneously; sequential use is safe.
- **`context-log.md` is the seam.** Append a short dated entry at the end of a working session (decisions made, files touched, next step) — the `dev-guidelines.md` already mandates this. Both tools read it, so it's how Kiro sees what Claude Code did and vice-versa.
- **Respect Kiro's conventions.** Specs live in `.kiro/specs/<feature>/{requirements,design,tasks}.md`; steering in `.kiro/steering/`. For multi-file features, follow the Kiro spec workflow rather than inventing a parallel structure. `AGENTS.md` is auto-generated by `n8nac` — don't hand-edit it.
- **Git:** branch before committing on `main`; commit/push only when asked.

## Core project rules (from steering — enforce these)

- **NO HARDCODING.** All user-facing config — slash commands, flags, themes, skills, **model lists** — must be DB-driven (Postgres), read via API. Adding a command/flag/model is a DB row, not a code constant. (Model IDs are currently hardcoded in a few spots; replacing that with dynamic Models-API discovery is the next planned task — see `project-review.md` §9.6.)
- Parameterized SQL always; never f-string/interpolate values into SQL. Identifiers that can't be parameterized go through the existing `_quote_ident` / validation helpers.
- Migrations are forward-only SQL files auto-applied on startup (`backend/database.py`). Note: the schema is **not currently rebuildable from scratch** — see `project-review.md` C2 before relying on a fresh DB.

## Dev environment (already set up — ready to run)

**Backend** (FastAPI, Python). A venv exists at `bowershub-ai/.venv` (Python 3.14; the Docker image targets 3.12, so treat local runtime parity as approximate). This system has **no `ensurepip`**, so the venv's pip was bootstrapped via the system pip — to add packages use:
```bash
cd bowershub-ai
/usr/bin/pip3 --python "$(pwd)/.venv/bin/python" install <pkg>   # if venv pip is ever missing again
.venv/bin/python -m pip install -r requirements.txt -r requirements-test.txt   # sync deps
```
Run tests (438 collect cleanly):
```bash
cd bowershub-ai
PYTHONPATH=. .venv/bin/python -m pytest -q                 # full suite — DB-backed tests need Postgres
PYTHONPATH=. .venv/bin/python -m pytest backend/tests/properties -q   # pure tests, no DB needed
```
DB-backed tests spin up ephemeral Postgres DBs and read `DB_HOST/DB_PORT/DB_USER/DB_PASSWORD` env vars (see `backend/tests/conftest.py`) — they require a reachable Postgres (the server's `ai-services_ai-network`), so they will error on a checkout with no DB.

**Frontend** (React + Vite + TS). Deps installed in `bowershub-ai/frontend/node_modules`:
```bash
cd bowershub-ai/frontend
npm run dev          # vite dev server (proxies /api and /ws to localhost:5003)
npm test             # vitest --run
npx tsc --noEmit     # typecheck (currently clean)
npm run build        # tsc && vite build
```

**Deploy** is on the server via `./scripts/deploy.sh <service>` or `docker compose up -d --build` per service dir. Don't deploy unless asked.

## Spec-driven feature planning (`/spec`)

This repo has a custom spec workflow that goes beyond Kiro's linear flow — invoke it with `/spec <feature-name>`. It authors `requirements.md` → `design.md` → `tasks.md` into `.kiro/specs/<slug>/` in **Kiro-compatible format** (so either tool can read/continue a spec), and it's multi-turn and human-gated (approve each phase before the next).

What makes it best-in-class (not a Kiro clone):
- **Grounded requirements** — fans out the `spec-researcher` subagent to map the real codebase + constraints before writing, so requirements reflect what exists.
- **Design tournament** (deep features) — spawns parallel design approaches (minimal-change / ideal-architecture / risk-first), then synthesizes the strongest.
- **Adversarial critique** — the `spec-critic` subagent hunts for gaps after each phase before you see it.
- **Mechanical traceability** — `.claude/hooks/spec-validate.py` enforces every requirement ↔ task before "done" (no LLM hand-waving).
- **Adaptive depth** — light / standard / deep; rigor scales to the feature.

Components: `.claude/skills/spec/` (SKILL.md + templates), `.claude/agents/{spec-researcher,spec-critic}.md`, `.claude/hooks/spec-validate.py`. A `PostToolUse` hook (`.claude/settings.json` → `.claude/hooks/spec-notify.py`) surfaces spec-file writes in the transcript (mirrors Kiro's `fileEdited`). **Note:** skills/agents/hooks load at session start — they take effect in a *new* session, not the one that created them.

Good first dogfood: `/spec dynamic-model-discovery` (the planned next task).
